# 🔬 深度研究报告：SWE-chat —— 编码 Agent 真实使用模式的首个大样本实证

> **论文**: Coding Agent Interactions From Real Users in the Wild (arXiv:2604.20779)  
> **作者**: Joachim Baumann 等（Stanford University）  
> **日期**: 2026-04-22  
> **数据**: https://huggingface.co/datasets/SALT-NLP/SWE-chat  
> **网站**: https://swe-chat.com  
> **代码**: https://github.com/SALT-NLP/SWE-chat

---

## 一、一句话总结

首个大规模真实编码 Agent 交互数据集揭示：**Agent 生成的代码仅 44% 进入最终提交，Vibe Coding 模式下的代码安全性比人类手写低 9 倍，且 Agent 在 44% 的交互回合中遭到用户"推回"**。

---

## 二、数据集规模

| 指标 | 数值 |
|------|------|
| 编码会话数 | ~6,000 |
| 用户提示数 | 63,000+ |
| Agent 工具调用数 | 355,000+ |
| 总日志事件 | 270 万 |
| 覆盖仓库数 | 200+ |
| 支持 Agent | Claude Code、OpenCode、Gemini CLI、Cursor、Factory AI Droid |

**数据采集方式**：开发者通过安装 Entire.io CLI 工具选择加入，自动记录会话日志并与代码提交关联（行级人类 vs Agent 作者归属）。

---

## 三、核心发现

### 3.1 使用模式：极端双峰分布

- **41% 的会话**：Agent 编写了几乎所有提交代码（"Vibe Coding"模式）
- **23% 的会话**：人类自己编写全部代码（完全不信任 Agent）
- 中间态极少——要么全交给 Agent，要么全自己动手

**解读**：开发者对 Agent 的态度呈现明显的"全有或全无"特征，缺乏渐进式信任构建。

### 3.2 代码效率：一半以上的 Agent 产出被丢弃

| 指标 | 数值 | 含义 |
|------|------|------|
| Agent 代码留存率 | **44%** | 超过一半的 Agent 产出不被信任 |
| 用户推回率 | **44%** | 通过纠正、失败报告和中断来修正 Agent 输出 |
| 用户中断率 | **5%** | 直接打断 Agent 的执行 |
| Vibe Coding token 消耗 | ~3× 协作模式 | 全自主模式的 token 和成本效率更低 |

### 3.3 安全性：Agent 代码引入更多漏洞

| 编码模式 | 每提交行的安全漏洞数（相对值） |
|---------|---------------------------|
| 人类手写 | 1.0×（基准） |
| 人机协作 | ~0.2× |
| Vibe Coding（全 Agent） | **~9×** |

**关键发现**：Vibe Coding 模式下的代码安全性最差，漏洞数是人类手写的 9 倍。

### 3.4 交互特征

- **最常见用户意图**：理解现有代码（而非写新代码）
- **Agent 工具调用分布**：1/3 的工具调用是执行 bash 命令（而非编辑文件）
- **最长自主执行时间**：99.9 百分位超过 100 分钟
- **Agent 主动询问率**：极低（长时间自主执行却很少主动澄清）

## 四、深层分析

### 4.1 为什么 Agent 代码留存率仅 44%？

1. **需求理解偏差**：Agent 倾向于过度实现或实现方向偏离
2. **代码风格不匹配**：生成代码与项目既有风格不一致
3. **安全盲区**：Agent 不主动考虑安全边界
4. **缺乏上下文**：虽有工具搜索，但对大型代码库的全局理解仍不足

### 4.2 为什么 Vibe Coding 安全性最差？

1. **无人审查**：完全自主模式下缺少人工 review 环节
2. **安全盲点**：Agent 对安全漏洞的敏感度低于人类开发者
3. **过度自信**：Agent 倾向于生成"能工作"的代码，而非"安全"的代码
4. **缺乏安全意识训练**：现有训练数据中安全标注不足

### 4.3 为什么用户推回率高达 44%？

1. **迭代式需求**：用户需求在交互过程中不断细化
2. **Agent 过度自主**：长时间不询问就执行，容易偏离方向
3. **质量不满意**：生成代码需要修正才能使用

---

## 五、对我们的启示

### 5.1 编码 Agent 使用策略

1. **强制安全审查**：对所有 Agent 生成的代码运行自动化安全扫描（semgrep、CodeQL）
2. **保留人工 review**：关键代码路径必须人工审查，不可完全信任 Agent
3. **渐进式委托**：从简单任务开始，逐步增加 Agent 自主权，而非一刀切
4. **监控代码留存率**：跟踪团队中 Agent 代码的实际采纳率，作为效率指标

### 5.2 Agent 系统改进方向

1. **增加主动澄清**：长时间自主执行前应主动询问用户确认方向
2. **安全意识注入**：在训练数据中增加安全标注，或在 prompt 中显式强调安全约束
3. **风格适配**：让 Agent 学习项目既有代码风格，减少风格冲突
4. **成本优化**：Vibe Coding 的 token 消耗是协作模式的 3 倍，需要平衡自主度和成本

### 5.3 研究价值

- **新基准构建**：SWE-chat 提供了超越精心策划 benchmark 的真实场景数据
- **用户模拟器**：可用于训练和评估 Agent 的用户交互能力
- **持续更新**：数据集是"活的"，会持续从公开仓库采集新会话

---

## 六、局限性与注意事项

1. **样本偏差**：仅覆盖选择加入 Entire.io 的开源开发者（早期采用者群体）
2. **Agent 分布**：约 85% 数据来自 Claude Code，不代表所有 Agent 的表现
3. **标注方式**：主要依赖 LLM-as-a-judge 进行标注，存在标注误差
