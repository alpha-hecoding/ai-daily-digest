## 二、Agent 架构与范式

### 1. SkillOS：自进化 Agent 的技能策展框架（arXiv:2605.06614）

**论文信息**：SkillOS: Learning Skill Curation for Self-Evolving Agents
- 作者：Siru Ouyang 等（Google Research, UIUC 等）
- 11 页，6 张图，3 张表格

**核心问题**：当前 LLM Agent 大多是「一次性问题解决者」，无法从过去的交互中学习。可复用的技能是自我进化的关键，但高质量技能策展（curation）是核心瓶颈。

**研究方法**：
1. **双组件架构**：冻结的 Agent 执行器（检索和应用技能）+ 可训练的技能策展人（从累积经验中更新外部 SkillRepo）
2. **组合奖励设计**：为策展提供学习信号
3. **分组任务流训练**：基于技能相关的任务依赖关系，早期轨迹更新 SkillRepo，后续相关任务评估更新效果

**核心发现**：
- 在多轮 Agent 任务和单轮推理任务上，SkillOS 一致超越无记忆基线和强记忆基线
- 学习的策展人能在不同执行器骨干和任务域之间泛化
- SkillRepo 中的技能随时间演化为更丰富结构的 Markdown 文件，编码更高层的元技能

**技术对比**：

| 方法 | 技能策展方式 | 长程学习 | 跨域泛化 |
|------|------------|---------|---------|
| 手动策展 | 人工编写 | 不支持 | 不支持 |
| 启发式操作 | 预定义规则 | 有限 | 有限 |
| 短期训练 | 短期优化 | 不支持 | 不支持 |
| **SkillOS** | **RL 学习策展** | **支持** | **支持** |

**💡 对你的价值**：如果你正在构建需要长期学习和自我改进的 Agent 系统（如客服 Agent、编程助手、数据分析 Agent），SkillOS 的思路非常有借鉴意义。关键 takeaway：技能应该被存储在结构化的 Markdown 文件中，随时间自动演化和精炼，而不是每次重新生成。

### 2. PrefixGuard：从 Agent 执行轨迹到在线失败预警监控器（arXiv:2605.06455）

**论文信息**：PrefixGuard: From LLM-Agent Traces to Online Failure-Warning Monitors
- 作者：Xinmiao Huang 等（Liverpool 大学等）

**核心问题**：LLM Agent 现在执行长时间、多工具的任务，等到最终结果检查时再干预已经太晚。需要在任务执行过程中就进行实时预警。

**研究方法**：
1. **StepView 归纳**：从原始轨迹样本中诱导确定性的类型化步骤适配器
2. **监控器训练**：学习事件抽象和前缀风险评分器
3. **DFA 提取**：事后提取紧凑的确定性有限自动机用于审计

**核心发现**：
- 在 WebArena 上达到 0.900 AUPRC，τ²-Bench 上 0.710，SkillsBench 上 0.533，TerminalBench 上 0.557
- 比原始文本控制平均提升 +0.137 AUPRC
- LLM 判定器在相同前缀预警协议下明显较弱
- DFA 在 WebArena 和 τ²-Bench 上保持紧凑（29 和 20 状态），但在 SkillsBench 和 TerminalBench 上扩展到 151 和 187 状态

**技术洞察**：强排名不意味着部署实用性。WebArena 排名好但无法支持低误报率预警，而 τ²-Bench 和 TerminalBench 保留了更多可操作的早期预警。

**💡 对你的价值**：如果你在生产环境部署 Agent 系统，PrefixGuard 提供了一套实用的监控合成方案。核心建议：不要依赖最终的成败判定来监控 Agent，而应该构建基于执行轨迹前缀的实时监控器，在 Agent 走偏之前就发出预警。

### 4. Fazm 博客揭示的 Agent 可靠性原则（综合 2026 年 5 月更新）

Fazm 的博客文章揭示了几个关键原则：

**Agent 调试与可靠性指南核心要点**：
1. **Agent 脚手架比模型更重要**：一个精心设计的系统提示中的八行代码可以比换模型带来更大的可靠性提升
2. **可访问性 API > 截图**：截图 Agent 丢失了六个关键信号，而可访问性树提供了 LLM 真正能读取的文本格式
3. **Token 经济学关键变量**：被忽视的变量是每轮输入（per-turn input），而非总 token 消耗
4. **Agent 模型供应商故障转移**：两天内实现的模型供应商故障转移模式

**💡 对你的价值**：如果你在使用或构建桌面 Agent（如 Mac 上的 AI 助手），这些原则直接适用。优先优化 Agent 的脚手架（系统提示、工具定义、错误恢复），然后才考虑换更大的模型。

### 2. Agent Token 经济学：被忽视的关键变量

**Fazm 博客揭示**：讨论 Agent 成本时，人们通常关注总 token 消耗，但真正关键的是**每轮输入（per-turn input）**。

**原因分析**：
- Agent 每轮都需要重新读取完整的对话历史
- 工具输出可能很长，大幅增加每轮输入
- 随着对话轮次增加，每轮输入呈线性增长

**优化策略**：
1. 压缩工具输出（只保留关键信息）
2. 使用摘要机制减少历史上下文
3. 在 Agent 脚手架中明确限制工具输出长度

**💡 对你的价值**：优化 Agent 成本时，不要只看模型大小。优化每轮输入长度（特别是工具输出的压缩）可以带来 30-50% 的成本下降。

### 1. ScaleLogic：RL 能否教会 LLM 长程推理？表达力是关键（arXiv:2605.06638）

**论文信息**：Tianle Wang 等

**研究方法**：
- 创建 ScaleLogic 合成逻辑推理框架
- 独立控制两个难度维度：证明规划深度（horizon）和底层逻辑表达力
- 支持从简单蕴含逻辑到包含合取、析取、否定、全称量化的复杂一阶逻辑

**核心发现**：
1. **RL 训练计算量 T 与推理深度 D 呈幂律关系**：T ∝ D^γ（R² > 0.99）
2. **缩放指数 γ 随逻辑表达力单调递增**：从 1.04（简单逻辑）到 2.60（复杂逻辑）
3. **在下游数学和通用推理基准上**：更具表达力的训练设置带来更大性能增益（最高 +10.66 分）和更高效的计算迁移
4. **幂律关系在多种 RL 方法上均成立**
5. **基于课程的训练大幅提升缩放效率**

**技术对比**：

| 逻辑表达力 | 缩放指数 γ | 计算需求增长 | 下游增益 |
|-----------|-----------|-------------|---------|
| 简单蕴含 | 1.04 | 线性 | 基准 |
| 含合取 | ~1.5 | 亚二次 | +5-7 分 |
| 完整一阶逻辑 | 2.60 | 超二次 | +10.66 分 |

**启示**：训练数据的表达力（质量）比训练量（数量）更重要。用更具表达力的逻辑进行训练，不仅带来更大的下游性能提升，而且在计算上更高效。

**💡 对你的价值**：如果你在微调模型做推理任务，不要盲目增加训练数据量。提高训练数据的逻辑复杂度（加入更多逻辑运算符和嵌套），能以更少的计算获得更好的效果。

### 2. POPO：超越负向采样——纯正向策略优化（arXiv:2605.06650）

**论文信息**：Mingwei Xu, Hao Fang

**核心问题**：当前主流的 GRPO 需要正负采样，但负向采样存在两个问题：无法区分失败的严重程度、组合空间太大导致稀疏二值奖励下有意义的信号很少。

**研究方法**：
1. **纯正向在线采样**：学习完全通过正向采样进行
2. **有界重要性采样**：在正向采样集上应用
3. **暹罗策略网络**：基于动量的自适应律实现稳定策略演化
4. **有界相似性惩罚**：在暹罗表示空间中替代 KL 散度

**核心发现**：
- 在数学基准上，POPO 达到与 GRPO 相当甚至更好的性能
- 关键指标：**POPO 在 AIME 2025 上以 Qwen-Math-7B 达到 36.67%**，超越 GRPO 的 30.00%
- 消融实验验证了 POPO 各组件的必要性和鲁棒性

**性能对比**：

| 方法 | 模型 | AIME 2025 | 需要负采样 |
|------|------|----------|-----------|
| POPO | Qwen-Math-7B | **36.67%** | 不需要 |
| GRPO | Qwen-Math-7B | 30.00% | 需要 |
| PPO | Qwen-Math-7B | ~25% | 需要 |

**💡 对你的价值**：如果你在做 LLM 的 RL 微调，POPO 提供了一种更高效的训练范式。纯正向采样不仅减少了计算量（不需要生成负样本），还带来了更好的效果。值得在你的项目中复现。
