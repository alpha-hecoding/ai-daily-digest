## 二、Agent 架构与范式

### 1. SWE-chat：首个大规模真实编码 Agent 交互数据集

**论文**：[arXiv:2604.20779](https://arxiv.org/abs/2604.20779)
**作者**：Joachim Baumann 等

**研究方法**
研究团队构建了 SWE-chat 数据集，这是第一个从真实开源开发者那里收集的编码 Agent 会话大规模数据集。目前已收录 6,000 个会话，包含超过 63,000 个用户提示和 355,000 次 Agent 工具调用。数据采集管道自动从公共仓库持续发现和收录会话。

**核心发现**
- **使用模式呈双峰分布**：41% 的会话中 Agent 编写了几乎所有提交代码（"vibe coding"模式），而 23% 的会话中人类自己编写全部代码
- **代码效率低**：Agent 生成的代码仅有 44% 最终进入用户提交——意味着超过一半的 Agent 产出被用户丢弃或重写
- **安全漏洞更多**：Agent 编写的代码比人类编写的代码引入更多安全漏洞
- **44% 的交互回合中用户在"推回"**：通过纠正、失败报告和中断来修正 Agent 输出

**对比分析**
| 指标 | 发现 | 含义 |
|------|------|------|
| Agent 代码留存率 | 44% | 超过一半的 Agent 产出不被信任 |
| 用户推回率 | 44% | Agent 输出需要频繁人工干预 |
| "Vibe coding"占比 | 41% | 接近一半用户完全依赖 Agent |
| 纯手工占比 | 23% | 仍有显著用户不信任 Agent |

**💡 对你的价值**：如果你正在使用编码 Agent，这条研究告诉你——不要盲目信任 Agent 的代码输出。建议：1）对所有 Agent 生成的代码做安全审查；2）保留人工 review 环节；3）关注 Agent 的错误模式，这数据将持续更新。

### 2. DR-Venus：仅用 10K 开放数据训练的前级边缘深度研究 Agent

**论文**：[arXiv:2604.19859](https://arxiv.org/abs/2604.19859)
**作者**：Venus Team（浙江大学等）

**研究方法**
研究团队探索如何在有限开放数据条件下训练强大的小型深度研究 Agent。DR-Venus 是一个 4B 参数的深度研究 Agent，完全基于开放数据构建。训练分为两个阶段：

1. **Agentic SFT 阶段**：建立基本 Agent 能力，结合严格数据清洗与长视野轨迹重采样，提升数据质量和利用率
2. **Agentic RL 阶段**：基于 IGPO 设计 turn-level 奖励，利用信息增益和格式感知正则化，增强监督密度和 turn-level 信用分配

**核心发现**
- DR-Venus-4B 在多个深度研究基准上显著优于此前所有 9B 以下参数的 Agent 模型
- 与 30B 级系统的差距大幅缩小
- 4B 模型已展现出令人惊讶的性能潜力
- 强调了测试时扩展（test-time scaling）在此场景下的价值

**💡 对你的价值**：如果你需要在边缘设备（如手机、IoT 设备）上部署 Agent 系统，4B 模型已经足够强大。关键不在于参数量，而在于训练策略——高质量数据 + 恰当的 RL 奖励设计比单纯增大模型更有效。

### 3. WorkflowGen：轨迹经验驱动的自适应工作流生成框架

**论文**：[arXiv:2604.19756](https://arxiv.org/abs/2604.19756)
**作者**：Ruocan Wei 等

**研究方法**
针对 LLM Agent 在复杂任务（如商业查询、工具使用、工作流编排）中面临的高推理开销、过多 token 消耗、不稳定执行和无法复用经验的问题，提出 WorkflowGen 框架。

核心机制：
- **轨迹捕获**：早期执行时捕获完整轨迹
- **知识提取**：在节点和工作流级别提取可复用知识，包括错误指纹、最优工具映射、参数模式、执行路径和异常规避策略
- **三层自适应路由**：根据与历史查询的语义相似度，动态选择直接复用、基于重写的生成或完整初始化

**核心发现**
- **token 消耗降低 40%+**（相比实时规划）
- **中等相似度查询成功率提升 20%**（通过主动错误规避和自适应回退）
- 无需大规模标注数据集

**💡 对你的价值**：如果你的 Agent 系统需要处理大量相似查询（如客服、数据分析管道），WorkflowGen 的思路可以直接借鉴——把过去的执行轨迹转化为可复用的"工作流模板"，大幅降低成本。

### 4. "工具过度使用幻觉"：为什么 LLM 偏爱外部工具而非内部知识？

**论文**：[arXiv:2604.19749](https://arxiv.org/abs/2604.19749)
**作者**：Yirong Zeng 等

**研究方法**
研究发现了一个关键但被忽视的现象：**工具过度使用**——即在推理过程中不必要地使用外部工具。通过两个视角分析其机制：

1. **知识认知幻觉**：模型错误判断内部知识边界，无法准确感知自身知识可用性。提出基于直接偏好优化（DPO）的知识感知认知边界对齐策略
2. **奖励结构因果关联**：可视化增强工具的练过程，发现"仅结果奖励"无意间鼓励了工具过度使用

**核心发现**
- DPO 策略将工具使用减少 **82.8%**，同时提升准确率
- 平衡训练中的奖励信号，在 7B 和 32B 模型上分别减少 **66.7%** 和 **60.7%** 的不必要工具调用，且不牺牲准确率

**💡 对你的价值**：如果你在训练 Agent 模型，这条研究非常重要——你的奖励函数可能在无意中鼓励 Agent 滥用工具。建议：在奖励设计中同时考虑工具使用效率和结果正确性。

### 5. NPO（Near-Future Policy Optimization）：利用"近未来的自己"加速 RL 训练

**论文**：[arXiv:2604.20733](https://arxiv.org/abs/2604.20733)
**作者**：Chenxu Yang 等

**研究方法**
提出了一种简单的混合策略方案：让策略从自己的"近未来版本"学习——同一训练运行中稍后的检查点作为辅助轨迹来源。这些轨迹既比当前策略更强，又比任何外部来源更接近当前分布。

**核心发现**
- 在 Qwen3-VL-8B-Instruct + GRPO 上，NPO 将平均性能从 **57.88 提升到 62.84**
- 自适应版本 AutoNPO 进一步提升至 **63.15**
- 加速了收敛速度，同时提高了最终性能天花板

**💡 对你的价值**：如果你在做 LLM 的 RL 后训练（如 GRPO、PPO），NPO 提供了一种简单但有效的技巧——不需要外部教师模型，利用自己训练过程中的检查点即可。

### 6. CFG 语法解析诊断：LLM 在 Agent 系统中的语法理解能力边界

**论文**：[arXiv:2604.20811](https://arxiv.org/abs/2604.20811)
**作者**：Hanqi Li 等

**研究方法**
引入 RoboGrid 框架，通过控制递归深度、表达式复杂度和表面风格的压力测试，分离评估 LLM 作为上下文解释器的语法、行为和语义三个维度。

**核心发现**
- LLM 通常能维持表面语法，但在结构密度高时（深层递归、高分支）性能崩溃
- 语义对齐在极端深度下完全消失
- CoT 推理只能部分缓解
- "Alien"词汇实验表明 LLM 依赖关键词语义引导而非纯符号归纳

**💡 对你的价值**：如果你在设计 Agent 的 DSL 或工具调用接口，不要假设 LLM 能可靠解析复杂语法。建议：保持语法扁平化，避免深层嵌套，为关键语法节点添加显式提示。

- **轻量级指令微调不损害基础模型的图像生成能力**

### 2. LLaDA2.0-Uni：扩散大语言模型统一多模态理解与生成

**核心发现**

**💡 对你的价值**：如果你需要同时做多模态理解（如视觉问答）和生成（如图像编辑），LLaDA2.0-Uni 提供了一个统一框架。代码和模型已开源：<https://github.com/inclusionAI/LLaDA2.0-Uni>
