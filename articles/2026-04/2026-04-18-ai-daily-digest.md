### 1. EvoMap/evolver：GEP 自进化引擎（737 ⭐/天）

**GitHub Trending #1** — [EvoMap/evolver](https://github.com/EvoMap/evolver)

**核心理念：** 使用 **基因组进化协议（Genome Evolution Protocol, GEP）** 让 AI Agent 自主进化。这是一个自进化引擎，通过模拟生物进化过程（变异、选择、交叉）来优化 Agent 的行为策略。

**技术细节：**
- **协议层**：GEP 定义了 Agent 能力的基因组表示、变异操作和适应度函数
- **自进化循环**：Agent 执行任务 → 评估表现 → 基因组变异 → 选择最优 → 下一代
- **技术栈**：JavaScript 实现，轻量级，易于集成
- **增长势头**：日增 737 星，总计 4.2K 星

**对比分析：**

| 特性 | EvoMap/evolver | GenericAgent | 传统 Agent |
|------|----------------|--------------|------------|
| 进化方式 | 基因组变异+选择 | 技能树生长 | 手动编程 |
| 自主性 | 高 | 高 | 低 |
| 可解释性 | 中（基因表示） | 高（技能树） | 高 |
| 适用场景 | 通用任务优化 | 系统化能力增长 | 特定任务 |

**💡 对你的价值：** 如果你正在构建需要持续改进的 AI Agent 系统，GEP 提供了一种"让 Agent 自己进化"的范式。适合长期运行的自动化任务，Agent 可以在执行过程中自我优化。

### 3. SuperLocalMemory V3.3："活体大脑"零 LLM Agent 记忆系统

**HuggingFace Papers 热门论文** — [SuperLocalMemory V3.3](https://superlocalmemory.com/)

**问题背景：** AI 编码 Agent 拥有海量参数知识，却无法记住一小时前说过的话。现有记忆系统仅将文本存储在向量数据库中，依赖云 LLM，缺乏人类记忆的认知过程。

**核心技术（五大贡献）：**

1. **Fisher-Rao 量化感知距离（FRQAD）**
   - 在高斯统计流形上定义的新度量
   - 100% 精度区分高保真和量化嵌入（对比 cosine 的 85.6%）
   - 零先验工作（全新方法）

2. **艾宾浩斯自适应遗忘**
   - Agent 记忆中首个数学遗忘曲线
   - 耦合渐进嵌入压缩，实现 6.7x 判别力提升

3. **7 通道认知检索**
   - 语义、关键词、实体图、时间、扩散激活、巩固、Hopfield 联想
   - 在 LoCoMo 基准上零 LLM 模式 A 达到 70.4%

4. **长期隐式记忆**
   - 通过 soft prompts 实现记忆参数化

5. **零摩擦自动认知流水线**
   - 自动化完整记忆生命周期

**性能表现：**

| 指标 | V3.3 (零LLM) | V3.2 (零LLM) | V3.2 (完整LLM) |
|------|-------------|-------------|----------------|
| LoCoMo | 70.4% | 74.8% | 87.7% |
| 多跳推理 | +23.8pp | — | — |
| 对抗性任务 | +12.7pp | — | — |

**运行要求：** 纯 CPU 运行，Elastic License 2.0 开源，月下载量 5,000+

**💡 对你的价值：** 如果你需要一个不依赖云端 LLM 的本地 Agent 记忆系统，SLM V3.3 是目前最成熟的方案。纯 CPU 运行意味着即使在低功耗设备上也能使用，特别适合隐私敏感场景。

### 4. MM-WebAgent：微软层级式多模态网页 Agent

**HuggingFace Papers** — [MM-WebAgent](https://aka.ms/mm-webagent)（Microsoft Research）

**问题：** 将 AIGC 工具直接集成到自动网页生成中，往往导致风格不一致和全局连贯性差，因为各元素是孤立生成的。

**解决方案：** 层级式 Agent 框架，通过层级规划和迭代自反思来协调 AIGC 元素生成。

**架构设计：**

```
MM-WebAgent 层级架构
├── 全局规划层（Global Planner）
│   ├── 页面布局设计
│   ├── 视觉风格统一
│   └── 元素关系建模
├── 局部生成层（Local Generator）
│   ├── 多模态内容生成（文本+图像）
│   ├── AIGC 工具协调
│   └── 风格一致性控制
└── 迭代自反思层（Self-Reflection）
    ├── 一致性检查
    ├── 冲突检测
    └── 自动修正
```

**性能：** 超越代码生成和基于 Agent 的基线，尤其在多模态元素生成和集成方面。

**💡 对你的价值：** 如果你在做自动化网页生成或设计系统，MM-WebAgent 的层级架构提供了一种解决"风格不一致"问题的有效模式。微软开源的代码和数据可以直接用于你的项目。

### 6. dflash：块扩散加速投机解码（287 ⭐/天，1.8K ⭐ 总计）

**GitHub Trending** — [z-lab/dflash](https://github.com/z-lab/dflash)

**概述：** DFlash 实现基于块扩散的投机解码（Speculative Decoding），大幅加速 LLM 推理。

**技术细节：**
- **投机解码**：使用小模型预生成 token，大模型验证

### 3. K-Token Merging：潜在空间序列压缩

**arXiv:2604.15153** — [Compressing Sequences in the Latent Embedding Space: K-Token Merging](https://arxiv.org/abs/2604.15153)

**问题：** LLM 处理长 prompt 时，全自注意力机制的计算和内存成本随输入长度呈二次方增长。现有 token 压缩方法主要在 token 空间操作，忽略了潜在嵌入空间的效率。

**K-Token Merging 方案：**
- 将每 K 个连续 token 的嵌入向量通过轻量编码器合并为单个嵌入
- 压缩后的序列由 LoRA 适配的 LLM 处理
- 生成阶段恢复到原始词表空间

**性能表现：**

|--------|----------|----------|----------|

| 50% | -1.2% | -0.8% | -1.5% |
| **75%** | **-3.5%** | **-2.1%** | **-4.2%** |

**💡 对你的价值：** 如果你的 LLM 应用经常处理长文档（>16K tokens），K-Token Merging 可以将输入长度压缩 75% 而只损失约 3-4% 的性能。这意味着处理速度和内存占用的大幅改善。

### 1. LLM 在最短路径问题中的泛化能力研究

**arXiv:2604.15306** — [Generalization in LLM Problem Solving: The Case of the Shortest Path](https://arxiv.org/abs/2604.15306)

**研究问题：** LLM 能否系统性泛化？这个问题一直有争议，因为经验性能受训练数据、训练范式和推理策略等多重因素影响。

**研究方法：**
- 构建基于最短路径规划的受控合成环境
- 这是一个典型的可组合序列化优化问题
- 两个正交泛化轴：**空间迁移**（未见过的地图）和**长度扩展**（更长路径）

**核心发现：**

| 泛化维度 | 结果 | 原因 |
|----------|------|------|
| 空间迁移 | ✅ 强 | 地图结构类似，模型可迁移 |
| 长度扩展 | ❌ 失败 | 递归不稳定性（recursive instability） |

**训练管道分析：**

| 阶段 | 作用 | 局限性 |
|------|------|--------|
| 数据覆盖 | 设定能力上限 | 覆盖率不足则能力受限 |
| 强化学习 | 提升训练稳定性 | 不扩展能力上限 |
| 推理时扩展 | 提升性能 | 无法挽救长度扩展失败 |

**启示：** 这项研究揭示了 LLM 在组合推理中的一个根本性弱点——可以处理"没见过的问题"（空间迁移），但无法处理"更长的同类问题"（长度扩展）。这对 Agent 系统设计有重要启示：不要让 LLM 处理超出其训练长度的序列推理任务。

**💡 对你的价值：** 如果你在设计需要多步推理的 Agent 系统，记住 LLM 的"递归不稳定性"问题。解决方案：将长链推理分解为多个短链子任务，或引入外部验证机制（如 SpecGuard）。
